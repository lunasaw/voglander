## §1 注册与认证

### 1.1 需求来源

GB/T 28181-2022 §7.1、§9.1：
- SIP UA 首次注册触发 `401 Unauthorized + WWW-Authenticate`，客户端 Digest 应答后 `200 OK`
- `Expires: 0` 注销设备
- 注册超时（心跳中断）自动离线
- 协议版本标识（`X-GB-Ver: 3.0`）

### 1.2 测试分层

| 测试类 | 层次 | 验证目标 |
|--------|------|----------|
| `SipRegisterE2eTest` | 真实协议 E2E | SIP 401→Digest→200 全链路 + DB |
| `DeviceRegistrationE2eTest`（已有） | 事件边界 E2E | Register 事件→异步→DB |

### 1.3 TDD 测试用例规格

#### TC-REG-01：完整注册握手 → 设备在线

```
前置条件：DB 无该设备记录
输入：ClientCommandSender.sendRegisterCommand(from, to, expires=3600)
期望：
  1. SIP 栈返回 200 OK
  2. tb_device.status = 1（ONLINE）
  3. tb_device.transport = "UDP"
  4. tb_device.register_expire = 3600
验证方式：Awaitility 5s 轮询 DeviceMapper
```

#### TC-REG-02：Expires=0 注销 → 设备离线

```
前置条件：TC-REG-01 已完成，设备 status=1
输入：ClientCommandSender.sendRegisterCommand(from, to, expires=0)
期望：
  1. SIP 栈返回 200 OK
  2. tb_device.status = 0（OFFLINE）
验证方式：Awaitility 3s
```

#### TC-REG-03：重复注册幂等性

```
前置条件：无
输入：连续发送 3 次相同设备注册
期望：
  1. tb_device 记录只有 1 条
  2. status = 1
  3. register_time 单调递增（最后一次覆盖）
```

#### TC-REG-04：密码错误 → 注册失败

```
前置条件：服务端 password=12345678
输入：客户端使用错误密码 wrongpass 发送 REGISTER
期望：
  1. DB 中无该设备记录（等待 2s 后确认）
注意：ClientCommandSender.sendRegisterCommand() 是单向发送，测试层无法获取 SIP 响应码，
      因此只通过 DB 状态判断；SIP 403 在 SIP 栈日志中可见但不作断言。
```

#### TC-REG-05：协议版本标识 3.0 兼容

```
前置条件：无
输入：REGISTER 消息头携带 X-GB-Ver: 3.0
期望：
  1. 注册成功 200 OK
  2. tb_device.gb_version = "3.0"（如字段存在）
```

class SipRegisterE2eTest {

                .eq(DeviceDO::getDeviceId, CLIENT_ID));

            assertThat(d.getStatus()).isEqualTo(1);
        });
    }

    @Test
    @DisplayName("TC-REG-02 Expires=0 注销 → 设备离线")

        ClientCommandSender.sendRegisterCommand(from(), to(), 0);
        await().atMost(3, SECONDS).untilAsserted(() ->

            assertThat(count).isEqualTo(1L);
            assertThat(deviceMapper.selectOne(Wrappers.<DeviceDO>lambdaQuery()
                .eq(DeviceDO::getDeviceId, CLIENT_ID)).getStatus()).isEqualTo(1);
        });
    }

    private boolean deviceOnline(String id) {
        DeviceDO d = deviceMapper.selectOne(Wrappers.<DeviceDO>lambdaQuery()
            .eq(DeviceDO::getDeviceId, id));
        return d != null && d.getStatus() == 1;
    }
}

## §2 保活与心跳

### 2.1 需求来源

GB/T 28181-2022 §9.6：设备周期发送 `MESSAGE` 保活（默认 60s），平台收到后响应 `200 OK`；超时未收到则标设备离线。

### 2.2 测试分层：事件边界 E2E

| TC | 验证目标 |
|----|----------|
| TC-KA-01 | Keepalive 事件 → 刷新 keepalive_time + status 保持 1 |
| TC-KA-02 | 连续 3 次心跳 → patchLivenessWithCoalesce 合并写保护 |

### 2.3 TDD 用例规格

#### TC-KA-01：心跳事件刷新保活时间

```
前置条件：设备已在线（status=1）
输入：注入 Notify.Keepalive 事件
期望：
  1. tb_device.keepalive_time 更新（>= 注入前时间戳 /1000）
  2. tb_device.status 仍为 1
验证方式：CountDownLatch(ShardDispatcher) + Awaitility 3s
```

#### TC-KA-02：快速连续心跳合并写

```
前置条件：设备在线，已有 5 条通道
输入：50ms 内连续注入 3 次 Keepalive 事件
期望：
  1. keepalive_time 取最新值
  2. tb_device_channel 通道数量不变（= 5）
```

### 2.4 测试类骨架

```java
// 文件：.../e2e/KeepaliveE2eTest.java
@Slf4j
@SpringBootTest(classes = ApplicationWeb.class,
    webEnvironment = SpringBootTest.WebEnvironment.DEFINED_PORT)
@ActiveProfiles("test")
class KeepaliveE2eTest {

    private static final String DEVICE_ID = "34020000001320000021";

    @Autowired VoglanderBusinessNotifier notifier;
    @Autowired DeviceMapper deviceMapper;
    @MockitoSpyBean ShardDispatcher shardSpy;

    @BeforeEach
    void registerDevice() {
        notifier.notify(new GatewayEvent("Lifecycle.Register", DEVICE_ID,
            null, System.currentTimeMillis(),
            Map.of("remoteIp","127.0.0.1","remotePort",5061,"expire",3600,
                   "localIp","127.0.0.1","transport","UDP"), "node-local"));
        await().atMost(5, SECONDS).until(() -> deviceOnline(DEVICE_ID));
    }

    @AfterEach
    void cleanup() {
        deviceMapper.delete(Wrappers.<DeviceDO>lambdaQuery()
            .eq(DeviceDO::getDeviceId, DEVICE_ID));
    }

    @Test
    @DisplayName("TC-KA-01 心跳事件刷新保活时间")
    void keepalive_refreshesTimestampAndKeepsOnline() throws InterruptedException {
        long beforeSec = System.currentTimeMillis() / 1000;
        CountDownLatch latch = new CountDownLatch(1);
        doAnswer(inv -> { latch.countDown(); return inv.callRealMethod(); })
            .when(shardSpy).dispatch(any());

        notifier.notify(new GatewayEvent("Notify.Keepalive", DEVICE_ID,
            null, System.currentTimeMillis(), Map.of("sn","1","status","OK"), "node-local"));

        assertThat(latch.await(5, SECONDS)).isTrue();
        await().atMost(3, SECONDS).untilAsserted(() -> {
            DeviceDO d = deviceMapper.selectOne(Wrappers.<DeviceDO>lambdaQuery()
                .eq(DeviceDO::getDeviceId, DEVICE_ID));
            assertThat(d.getKeepaliveTime()).isGreaterThanOrEqualTo(beforeSec);
            assertThat(d.getStatus()).isEqualTo(1);
        });
    }

    private boolean deviceOnline(String id) {
        DeviceDO d = deviceMapper.selectOne(Wrappers.<DeviceDO>lambdaQuery()
            .eq(DeviceDO::getDeviceId, id));
        return d != null && d.getStatus() == 1;
    }
}
```

## §3 设备目录查询

### 3.1 需求来源

GB/T 28181-2022 §9.5.1：平台向设备发送 `MESSAGE`（CmdType=Catalog），设备响应目录列表（含 SumNum 分页）；平台收到后写入 `tb_device_channel`。

### 3.2 测试分层

| TC | 层次 | 验证目标 |
|----|------|----------|
| TC-CAT-01 | 事件边界 E2E | Catalog 响应 → 批量 upsert 10 条通道 |
| TC-CAT-02 | 事件边界 E2E | 分批 2×10 → 合并后 20 条，无重复 |
| TC-CAT-03 | 事件边界 E2E | 重复上报 → 幂等，记录数不变 |
| TC-CAT-04 | 事件边界 E2E | 失踪通道 → markMissing → status=0 |
| TC-CAT-05 | Manager 集成 | batchUpsertWithStatus 单调性保护 |

### 3.3 TDD 用例规格

#### TC-CAT-01：目录响应 → 批量写通道

```
前置条件：设备在线，无通道记录
输入：注入 Response.Catalog，deviceList=[10 条]
期望：
  1. tb_device_channel 新增 10 条，device_id 全部匹配
  2. channel_status = 1（ONLINE）
验证方式：Awaitility 5s
```

#### TC-CAT-03：幂等写保护

```
前置条件：已有 5 条通道
输入：重复注入相同 Catalog 事件 3 次
期望：tb_device_channel 仍为 5 条
```

#### TC-CAT-04：失踪通道标离线

```
前置条件：通道 [ch01..ch10] status=1
输入：新 Catalog 事件只包含 [ch01..ch07]
期望：ch08..ch10 的 channel_status = 0
```

#### TC-CAT-05：Manager 层单调性保护

```
测试类：DeviceChannelManagerTest extends BaseTest
方法：batchUpsertWithStatus_shouldIgnoreOlderTimestamp
前置：upsert 一条通道，update_time = T+10
输入：再次 upsert 相同通道，update_time = T
期望：DB 记录 update_time 仍为 T+10（不回退）
```

        return new GatewayEvent("Response.Catalog", deviceId,
            null, System.currentTimeMillis(),
            Map.of("sumNum", channels.size(), "deviceList", channels), "node-local");
    }

    private GatewayEvent buildAddEvent(List<String> channelIds) {
        List<Map<String,Object>> list = channelIds.stream()
            .map(id -> (Map<String,Object>) Map.of("channelId", id, "name", id,
                "status", "ON", "event", "ADD"))
            .toList();
        return new GatewayEvent("Response.Catalog", DEVICE_ID,
            null, System.currentTimeMillis(),
            Map.of("sumNum", list.size(), "deviceList", list), "node-local");
    }
}

## §5 实时视音频点播

### 5.1 需求来源

GB/T 28181-2022 §9.2：三方呼叫控制（第三方发起），流程：
`INVITE → 100 Trying → 200 OK（SDP） → ACK → RTP 流传输 → BYE → 200 OK`

### 5.2 TDD 用例规格

| TC | 层次 | 验证目标 |
|----|------|----------|
| TC-RTP-01 | 事件边界 E2E | InviteOk 事件 → MediaSession 入库（status=1/ACTIVE，sessionType=PLAY） |
| TC-RTP-02 | 事件边界 E2E | Bye 事件 → MediaSession status=0（CLOSED） |
| TC-RTP-03 | 事件边界 E2E | InviteFailure 事件 → status=3（FAILED） |
| TC-RTP-04 | Manager 集成 | MediaSessionManager.create + close 事务正确 |

#### TC-RTP-01：实时点播成功 → MediaSession 在线

```
前置条件：设备在线，通道存在
输入：注入 Session.InviteOk，携带 callId/ssrc/streamUrl
期望：
  1. tb_media_session 新增记录，status = 1（ACTIVE），sessionType = "PLAY"
  2. stream 字段非空
验证：Awaitility 5s
```

#### TC-RTP-02：BYE → MediaSession 关闭

```
前置条件：TC-RTP-01 完成，session status=1
输入：注入 Session.Bye，callId 同上
期望：tb_media_session.status = 0（CLOSED）
```

#### TC-RTP-03：邀请失败记录

```
前置条件：通道存在
输入：注入 Session.InviteFailure，携带 callId + errorCode=503
期望：tb_media_session.status = 3（FAILED）
```

## §17 新增陷阱与解法

> 基于 GB28181-2022 全协议测试预估，补充 1.0.4 模板未覆盖的陷阱。

### 陷阱 6：级联事件与 Spring `ApplicationEvent` 混淆

**现象**：`CascadeClientRegisterListener` 监听的是 Spring `ApplicationEvent`，而 `DeviceRegistrationE2eTest` 使用 `VoglanderBusinessNotifier.notify(GatewayEvent)`，两者发布机制不同。

**根因**：Spring `ApplicationEvent`（`publishEvent`）与 GatewayEvent 走不同的分发链路。

**解法**：级联测试用 `ApplicationEventPublisher.publishEvent()` 发布 `ClientRegisterSuccessEvent`（来自 `io.github.lunasaw.gbproxy.client.eventbus.event`）；设备注册测试用 `VoglanderBusinessNotifier.notify()` 注入 `GatewayEvent`。不要混用。

---

### 陷阱 7：ZLM Mock 注入失败（多 Bean 候选）

**现象**：`@MockitoSpyBean ZlmApiWrapper zlmSpy` 报 `No qualifying bean`。

**根因**：ZLM 封装类可能通过工厂方法创建，不是直接 `@Component`。

**解法**：改用 `@MockitoBean` 完全替换，或在 `CacheTestConfig` 中注册 `@TestConfiguration` 替代 Bean（注意 §4 陷阱 1 的 Profile 隔离要求）。

---

### 陷阱 8：Awaitility 超时误报（事件 type 字符串不匹配）

**现象**：注入 GatewayEvent 后 Awaitility 始终等不到 DB 变化，超时失败。

**根因**：`Gb28181ProtocolHandler` 的 switch 分支按 `groupName` 匹配，`groupName` 是 `GatewayEvent.type()` 去掉协议前缀后的部分。`VoglanderBusinessNotifier` 解析规则：`"Lifecycle.Register"` → protocol=`"Lifecycle"`, groupName=`"Lifecycle.Register"`。事件 type 字符串必须是 `"Xxx.Yyy"` 格式，**不含** `"gb28181."` 前缀。

**正确格式示例**：
- ✅ `"Lifecycle.Register"` / `"Notify.Keepalive"` / `"Response.Catalog"` / `"Session.InviteOk"`
- ❌ `"gb28181.Lifecycle.Register"` / `"gb28181.Notification.Keepalive"`

**解法**：用 `@MockitoSpyBean ShardDispatcher` 的 `CountDownLatch` 确认 dispatch 被调用；若未调用则对照 `Gb28181ProtocolHandler` 的 switch 分支检查 type 字符串。

---

### 陷阱 9：MediaSession 状态断言使用字符串（实际是整型）

**现象**：`assertThat(s.getStatus()).isEqualTo("PLAYING")` 编译通过但断言失败，或产生类型不匹配错误。

**根因**：`MediaSessionDO.status` 是 `Integer` 类型，枚举值为：`0=CLOSED`、`1=ACTIVE`、`2=INVITING`、`3=FAILED`。会话类型（直播/回放/下载/广播）由独立的 `sessionType` 字段（`String`）区分：`PLAY / PLAYBACK / DOWNLOAD / TALK`。

**解法**：
```java
// 实时点播
assertThat(s.getStatus()).isEqualTo(1);
assertThat(s.getSessionType()).isEqualTo("PLAY");
// 录像回放
assertThat(s.getStatus()).isEqualTo(1);
assertThat(s.getSessionType()).isEqualTo("PLAYBACK");
// 语音广播
assertThat(s.getStatus()).isEqualTo(1);
assertThat(s.getSessionType()).isEqualTo("TALK");
// BYE 后
assertThat(s.getStatus()).isEqualTo(0);
// 失败
assertThat(s.getStatus()).isEqualTo(3);
```

---

### 陷阱 10：PTZ 命令未找到设备 → NullPointerException

**现象**：`commandService.ptzControl()` 抛 NPE，而非业务异常。

**根因**：`DeviceAgreementService.getCommandService(deviceId)` 在设备不存在时返回 null，未做 null 检查。同时 `ptzControl` 接受 `DevicePtzReq` 对象，而非 `(deviceId, channelId, direction, speed)` 四参数形式。

**解法**：E2E 测试须在 `@BeforeEach` 确保设备在线（先注入 `Lifecycle.Register` 事件，等待 Awaitility 确认）；调用时构造 `DevicePtzReq` 对象传入。

---

### 陷阱 11：`tb_alarm` 表不存在 → 告警 TC 无法 DB 断言

**现象**：`AlarmE2eTest` 中查询 `tb_alarm` 表报 `no such table`。

**根因**：当前版本 `Notify.Alarm` 事件进入 `Gb28181ProtocolHandler` 后仅做日志记录，无 `AlarmMapper`/`AlarmDO`，`schema-sqlite.sql` 中也无 `tb_alarm` 表。

**解法**：TC-ALM-01/02/03 当前阶段改为验证 `ShardDispatcher.dispatch()` 被调用（CountDownLatch 方式），即 TDD Red 阶段；待 `tb_alarm` 落库逻辑实现（Sprint 2）后，在测试中追加 DB 断言并移除 TODO 注释。

---

### 陷阱 12：`CascadePlatformManager.add()` 必填字段校验失败

**现象**：`add()` 抛 `IllegalArgumentException: platformIp 不能为空`。

**根因**：`CascadePlatformManager.add(CascadePlatformDTO)` 内有 `Assert.hasText` 校验，要求 `platformIp` 和 `localClientId` 非空，而方案骨架中仅设置了 `platformId`。

**解法**：`@BeforeEach` 中必须同时设置：
```java
dto.setPlatformIp("127.0.0.1");
dto.setLocalClientId(PLATFORM_ID);
```
