# 1.0.9 图像资产与采集运维说明

## 启用与回滚

`voglander.image.enabled=true` 时，启动检查必须发现通用 durable business task engine 和唯一 `IMAGE_COLLECTION` Handler。发布顺序为：先执行对应数据库的 durable-task migration，再执行 `1.0.9-image-asset-collection` migration，最后启用图像 API。回滚代码时保留三张图像表和 migration key；先关闭图像 API/Handler，再回滚应用，不执行 `DROP TABLE`。

## 本地存储

本地 provider 的 canonical root 由 `voglander.image.storage.root` 指定；每个 worker 使用 `.staging/{workerNode}` 临时目录。文件只能通过 `ImageStorageService` 读写，不能把 root 暴露为静态目录。生产多实例必须使用共享卷，或保证每个 worker 能访问所有 final key；reconciliation 默认 `report-only`。

## 任务与故障处理

任务、执行、事件、租约、进度和控制统一由 durable task engine 负责。图像 Handler 只处理设备/通道在线校验、媒体 lease、ZLM snapshot、验证、stage/promote 和 completion participant。截图、临时节点、临时存储失败可重试；权限、机位不存在、配置和图像验证失败永久失败。DB 提交失败时删除 final object；补偿失败由 orphan recorder 记录并由 reconciliation 报告。

## 观测

图像域指标：`image_collection_handler_total`、`image_snapshot_duration_seconds`、`image_storage_operation_duration_seconds`、`image_storage_orphan_total`、`image_storage_reconciliation_total`。任务 lag、queue、lease 指标使用通用任务指标。日志只记录脱敏 code 和稳定业务标识，禁止 token/JWT、secret、URL query、路径和原始 provider 错误。

## 验证清单

- SQLite migration 在已有 core migration 的库上可重复执行，并保留已有数据。
- MySQL/PostgreSQL 在目标环境执行全量与增量脚本，保存表、索引、唯一键和菜单查询结果。
- 真实 ZLM POC 记录 RTSP/HTTP-FLV/HLS URL 优先级、`timeout=15`、`expire=0`、savePath 清理及 stopLive 引用归还。
- 前端通过资产 SSE 与通用任务 SSE 触发 debounce refresh；断线后全量查询。
